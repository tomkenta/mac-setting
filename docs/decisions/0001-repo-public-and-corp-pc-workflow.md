# ADR-0001: リポジトリの public 化と「会社PC ⇄ 個人 repo」運用方針

- ステータス: 実施済み (Accepted)
- 日付: 2026-06-19
- 対象リポジトリ: `tomkenta/mac-setting`

> 注: 本ドキュメントには、公開履歴から除去した前職由来の識別子（社用メールアドレス・
> 社内ホスト名など）の**実値は意図的に記載しない**。実値を書くと除去した意味が失われるため、
> 一般化した表現のみを用いる。

## 背景 / 目的

会社の PC から、この Mac キッティング設定 (`mac-setting`) を**参照**してセットアップに使いたい。
そのために repo を private → **public** へ変更したい。

public 化にあたり、以下を懸念点として確認した。

- コードに機密情報（トークン・パスワード・鍵）が含まれていないか
- git 履歴（削除済みファイル・過去コミット含む）に機密情報が残っていないか
- 「会社 PC から個人 repo を扱う」こと自体のリスク

## 事前監査の結果

公開前に、コードと **到達可能な全 git オブジェクト（現行＋削除済み＋バイナリ）** を網羅スキャンした。

- **シークレット／認証情報: 検出 0 件**
  - GitHub / Slack / OpenAI・Anthropic / AWS / Google などの代表的なトークン形式、各種
    PRIVATE KEY、JWT、`ANSIBLE_VAULT`、URL 埋め込み `user:pass@` を全 blob に対して
    総当たり → すべて 0 件
  - 廃止済みの Ansible 構成（`group_vars` / `inventory/localhost` / `ansible.cfg` /
    `localhost.yml` / `roles/*` / 旧 `setting.sh`）も全バージョンを確認 → パッケージ一覧・
    公開リポジトリの clone・ローカルホスト IP のみ。vault・パスワード・トークンなし
  - README のトークン項はプレースホルダ、Alfred の 1password 設定は空、バイナリ
    （`.data` / `.png`）にもトークンなし

- **唯一の指摘: git メタデータに前職由来の識別子**
  ファイル内容ではなく **コミットの著者／コミッタ情報**に、前職由来の識別子が 2 コミットだけ
  残っていた。どちらも master の系譜に含まれ、public 化すると閲覧可能だった。

  | 旧コミット | 内容 | 残存していた識別子（実値は伏せる） |
  |---|---|---|
  | initial commit | `initial commit` | 前職の社用メールアドレス（committer） |
  | fisherman role 修正 | `[MOD] fix syntax error for fisherman role` | 前職の社内ホスト名を含むアドレス（author/committer） |

  認証情報ではないが、**前職の社用メールと社内マシン名が公開履歴に永続的に残る**ため、公開前に
  処理する価値があると判断。氏名と個人用メール (`tomkenta2014@gmail.com`) は既に公開済みの
  ため、新規に露出し得るのは上記の前職由来識別子のみだった。

  > 補足: なぜ混入したか = 当時、会社環境（会社 PC / 社内 identity）でコミットしたため、git が
  > 社用メール・社内ホスト名を author/committer に焼き付けた。これは本 ADR の運用方針の根拠でもある。

## 決定

### 決定 1: 編集は個人マシンに一本化（会社 PC は読み取り専用）

会社 PC からは **clone / pull（参照）のみ**。**編集・commit・push は個人マシンからのみ**行う。

**理由（会社 PC から個人 repo へ push することのリスク）:**

1. 会社のコード／機密を意図せずコミットに巻き込む事故（DLP・利用規程違反）
2. コミットに会社の identity（社用メール・社内ホスト名）が焼き付く ← 今回の前職由来メタデータが該当
3. 個人の認証情報（SSH 鍵 / PAT）を会社端末に置くことになる（監視対象・退職時に喪失）

**会社 PC で見つけた設定を個人設定へ反映したい場合（運用）:**

- 設定そのものを push せず、**変更内容だけ**を持ち帰る（設定値・パッケージ名・キーバインドは
  自分の好みの再宣言にすぎない）。会社 PC ではブラウザから Issue/メモを立て、個人マシンで再現して
  コミットする。
- どうしても会社 PC で直接コミットする場合は、最低限その clone 内で identity を明示設定し、
  社内ホスト名の焼き付きを防ぐ:

  ```bash
  git config user.name  "tomkenta"
  git config user.email "tomkenta2014@gmail.com"
  ```

  （`user.email` を明示設定すると `名前@ホスト名` の自動生成が止まり、社内ホスト名は入らない）
  その上で専用ブランチへ push し、毎回 diff を確認し、master へのマージは個人マシンで行う。

### 決定 2: 公開前に前職由来メタデータを履歴から除去

「そのまま公開」ではなく、**public 化の前（private のまま）に履歴を書き換える**ことを選択。
これにより前職の社用メール／社内ホスト名は一度も公開されずに消える。Issue（30 件）と PR は
GitHub 側に保持され影響を受けない。

### 決定 3: repo を public 化

監査と履歴書き換えの完了後に visibility を public へ変更。

## 実施記録

1. **バックアップ**: 全 ref を bundle 化（リポジトリ外のホームディレクトリに保存）。
   ※このバックアップには書き換え前（前職由来識別子を含む）履歴が入るため、保管・廃棄に注意。
2. **不要ローカルブランチ削除**: リモートで既に削除済みだった `add-ci` /
   `modernize-for-apple-silicon` / `remove-ansible` のローカル残骸を削除
3. **履歴書き換え**: `git filter-repo --mailmap` で該当 2 コミットの author/committer を
   `tomkenta <tomkenta2014@gmail.com>` に置換（dry-run で前職由来識別子が 0 件になることを
   確認後に本実行）。initial commit を触ったため全コミットの SHA が変化。
4. **force-push（private のまま）**: 実在する 3 ブランチ `master` /
   `add-claude-statusline` / `sync-current-configs` を更新。旧コミット（前職由来識別子入り）は
   到達不能化
5. **検証**: origin 全 ref を前職由来の識別子（社用メールドメイン・社内ホスト名・社内ドメイン）で
   再検索し 0 件を確認
6. **public 化**: `gh repo edit --visibility public`

## 影響 / 注意点

- 履歴書き換えにより **全コミットの SHA が変わった**（initial commit を触ったため）。
  既存のローカルクローンがある場合は re-clone 推奨。
- マージ済み PR ページが参照する旧 SHA は到達不能になり得る（PR 本文・コメントは残る）。
  なお前職由来識別子を含むコミットはどの PR からも参照されていない（2020 年のベース履歴）。
- `add-claude-statusline` / `sync-current-configs` は master にマージ済みで内容は冗長。
  必要に応じて削除してよい（本対応では現状維持）。

## 教訓

- **記録（ADR）に機密の実値を書かない。** 一度、本 ADR に前職の社用メール・社内ホスト名の実値を
  そのまま記載して push してしまい、履歴除去の意味を損なった。実値は一般化表現に置き換え、
  当該コミットを書き換えて除去した。「何を除去したか」は型・分類で記録すれば十分で、実値は不要。

## 今後のルール（要約）

- 会社 PC = 参照（clone/pull）のみ。編集・push は個人マシンから。
- 会社 PC で commit する場合は repo ローカルで個人 identity を明示し、diff を必ず確認。
- 会社の機密・コードを個人 repo に絶対に入れない。
- 個人 repo に push する内容は「自分の設定（好み）」に限定する。
- ドキュメント・コミットメッセージにも機密の実値を書かない（一般化して記録する）。
